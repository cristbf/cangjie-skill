# cangjie-skill 改进计划

> 基于对 RIA-TV++ 流水线的深度分析，提出的系统性改进方案。

## 总体评价

RIA-TV++ 的核心洞察正确："为 Agent 而非人类读者蒸馏"。流水线设计在 Prompt 工程领域属于少见的"有方法论自觉"的作品。

**能做好的部分**：显式的、结构化的、可操作的方法论（约覆盖一本好书 60-70% 的价值）。

**需要补充的部分**：隐含在叙事中的智慧、多步推导的理论、作者自己也没意识到的洞察。

## 改进分层

```
层次 1: 修复现有缺陷（质量提升）
层次 2: 增加缺失维度（覆盖盲区）
层次 3: 适配不同类型书籍（泛化能力）
```

---

## 层次 1：修复现有缺陷

### 1.1 改造 V3（独特性检验）— 最高优先级

**问题**：V3 问"这是不是常识"，但"常识"没有锚点，高度主观。AI 对"常识"的判断 ≠ 人类的常识，可能误杀有价值的朴素方法论。

**方案**：把 V3 从凭感觉判断改为**有锚点的对照验证**。

- 新增 `v3-baseline-extractor.md`，在 Stage 1.5 之前先跑一轮
- 用 AI 生成该领域 3 本同类书对同一问题的典型回答（不需要真读过，用训练集知识即可）
- V3 问题变为："这本书的回答和同类书的典型回答有显著差异吗？"
  - 有差异 → 通过（作者的独特洞察）
  - 无差异 → 检查是否只是复述常识
  - 差异但方向相反 → 特别标记（可能是最有价值的反直觉洞察）

### 1.2 改造 V2（预测力测试）— 高优先级

**问题**：同一个 AI 设计场景又评判答案，有循环论证风险。

**方案**：拆成两步，引入对抗性场景。

- **场景设计**：不许提取者自己设计新问题，改为从两个来源获取：
  - 用户提供的"我实际遇到的问题"（Stage 0 时顺带收集）
  - 独立的 adversarial agent 设计故意刁难的场景（如"这个方法论在什么情况下会彻底失效？"）
- **答案评判**：标准从"有没有意义"改为：
  - 推导过程是否**可复现**（换个人按同样步骤能得出类似结论吗）
  - 结论是否**可证伪**（有没有办法检验对错）
  - 是否**非平凡**（不是"努力就会成功"这种废话）

### 1.3 引用限制分段化 — 中优先级

**问题**：150 字上限对多步推理型方法论是灾难，压缩过程本身就是信息丢失。

**方案**：改为分段引用 + 总量上限。

- R 段允许引用 2-4 个片段，每个 ≤150 字
- 每个片段标注 `章节:段落` 或 `页码:行范围`
- 片段之间用 `...` 或 `[此处省略论证过程]` 连接
- 总引用量上限 500 字
- 模板 R 段格式：

```markdown
## R — 原文 (Reading)

> [片段 1] "反过来想,总是反过来想..."
> — 第 3 讲

> [片段 2] "如果知道我会在哪里死去,那我就永远不去那里。
> 这意味着,第一步不是想怎么成功,而是列出所有可能让我
> 死掉的原因..."
> — 第 3 讲

> [片段 3] "在投资中,逆向思维意味着先问'什么会让我
> 亏光',而不是'什么能让我赚十倍'..."
> — 第 7 讲
```

## 层次 2：增加缺失维度

### 2.1 新增"未解决的张力"维度 — 高优先级

**问题**：只提取作者声称有效的东西，忽略了知识的"负空间"（矛盾、局限、开放问题）。

**方案**：在 Stage 0 的 Adler 批判阶段增加输出字段，并新增第 6 个提取器。

Stage 0 增加输出：

```markdown
## 内在张力与未解问题

### 作者未解决的矛盾
- 矛盾 1: 作者在第 3 章强调 X，但在第 8 章的实际案例中做了非 X

### 作者承认的局限
- "这个方法在 Y 场景下可能不适用"（第 12 章原文）

### 开放问题
- 作者提出了问题但没给出答案: "..."
```

新增 `tension-extractor.md`（第 6 个提取器）：
- 搜索作者自相矛盾的地方
- 搜索作者明确说"我不确定"的段落
- 搜索作者引用了但没深入讨论的反对观点
- 这些张力不做成独立 skill，写入相关 skill 的 B 段作为内在局限

### 2.2 新增"反对者视角"维度 — 中优先级

**问题**：蒸馏出的 skill 只呈现作者的"胜利叙事"，缺乏对抗性视角。

**方案**：在 Stage 0 增加 Step 4.5: 外部批评扫描。

- 让 AI 基于训练集知识列出：
  - 这本书的主要学术/行业批评
  - 对立学派的核心反驳论点
  - 书中方法论在实证研究中的表现（如果有已知数据）
- 输出到 `BOOK_OVERVIEW.md` 新段落
- 信息流入两个地方：
  - skill 的 B 段（此方法论的已知反对意见）
  - Stage 3 的 Zettelkasten 链接（增加 `contrasts-with-external` 关系类型）

### 2.3 用户反馈闭环 — 中优先级

**问题**：蒸馏 → 测试 → 交付，单向流动，没有回流机制。

**方案**：定义 `feedback.json` 格式，建立使用信号回流。

模板 `templates/feedback.json.template`：

```json
{
  "skill": "inversion-thinking",
  "feedback_entries": [
    {
      "date": "2026-05-29",
      "trigger_type": "correct | missed | false_alarm",
      "user_query": "用户的原始问题",
      "expected_trigger": true,
      "actual_triggered": true,
      "output_quality": "good | acceptable | poor",
      "issue": "如果质量差，具体是什么问题",
      "suggestion": "用户或 AI 的改进建议"
    }
  ]
}
```

新增可选 **Stage 5 — 反馈整合**：
- 收集 N 条反馈后分析模式
- `false_alarm` 多 → 收紧 A2 的 trigger 描述
- `missed` 多 → 放宽 A2 或增加 trigger 信号
- `output_quality: poor` 多 → 检查 E 段步骤是否不够具体
- 更新 SKILL.md，版本号 +1

## 层次 3：适配不同类型书籍

### 3.1 新增"书籍类型判断"入口 — 最高优先级

**问题**：流水线对所有书一视同仁，但不同类型书籍需要不同处理策略。

**方案**：在 Stage 0 之前增加 Pre-Stage: 书籍类型分类。

```
阶段 -1: 书籍类型判断
读取书名、目录、前言，判断属于以下哪种类型：

A. 框架型（商业/管理/投资/决策）  → 标准 RIA-TV++ 流水线
B. 叙事型（历史/传记/纪实）      → 调整参数（见 3.2）
C. 理论型（哲学/科学/学术）      → 调整参数（见 3.3）
D. 体验型（文学/散文/诗歌）      → 跳过或用替代方案（见 3.4）
```

判断结果写入 BOOK_OVERVIEW.md 头部，后续阶段据此调整参数。

### 3.2 叙事型书籍适配 — 中优先级

**问题**：历史/传记的方法论隐含在叙事中，没有显式的"框架"或"原则"段落。

**方案**：
- framework-extractor 改为 pattern-extractor：不只找作者命名的框架，还找反复出现的行为模式
- V1 调整：跨域验证从"书内 2 个独立段落"改为"2 个独立事件/时期/案例"
- 新增隐含方法论提取信号：
  - 作者没有明确命名，但同一模式出现 3 次以上
  - 人物在关键时刻的决策过程
  - 事后被证明正确的反直觉选择

### 3.3 理论型书籍适配 — 低优先级（复杂度高）

**问题**：哲学/科学书的论证是多步推导链，不能原子化成独立 skill。

**方案**：
- 放宽引用限制：理论型书籍 R 段总量上限提升到 800 字，允许保留完整推导链
- 新增 `argument-chain-extractor.md`：提取从前提到结论的推理路径
- RIA++ 结构调整：理论型 skill 的 E 段写"论证复现步骤"而非操作步骤
- V2 调整：预测力测试改为"能否用这个推理路径分析一个新的论题"

### 3.4 体验型书籍的替代方案 — 低优先级

**问题**：文学/诗歌的知识是体验性的，不能提取成可执行方法论。

**方案**：
- 在 SKILL.md 边界中明确：体验型书籍不适用 book2skill
- 提供替代建议：此类书籍建议使用 nuwa-skill（蒸馏作者的思维风格和表达 DNA）
- 如果用户坚持，可提取写作技巧（叙事结构、修辞手法、节奏控制），但属于另一个 skill 类型

## 前置验证：skill 实际触发率测试

在动手改框架之前，有一个更根本的问题需要先回答：

**Claude Code 的 skill 激活机制，实际命中率有多高？**

RIA-TV++ 的所有质量门控最终都落在这个问题上：当用户说出一个该触发某 skill 的问题时，Claude 真的会触发它吗？

**建议**：用现有的 200+ 个已产出 skill 做一次实际触发率测试 — 随机抽 20 个 skill，各设计 5 个应该触发的真实场景，看 Claude 实际触发了多少。这个数据会决定改进的重心在哪里。

## 评分总结

| 维度 | 评分 | 说明 |
|------|------|------|
| 核心洞察 | ⭐⭐⭐⭐⭐ | "为 Agent 而非人类读者蒸馏" — 原创且正确 |
| 流水线设计 | ⭐⭐⭐⭐ | 6 阶段逻辑清晰，质量门控层层递进 |
| 三重验证 | ⭐⭐⭐⭐ | V1 和 V2 很锋利，V3 有主观性问题 |
| RIA++ 结构 | ⭐⭐⭐⭐⭐ | A2（触发场景）是真正的创新 |
| 可操作性 | ⭐⭐⭐⭐ | 模板和提取器定义完整，AI 可直接执行 |
| 鲁棒性 | ⭐⭐⭐ | 对框架类书籍好，对叙事/理论类书籍弱 |
| 闭环能力 | ⭐⭐⭐ | 有 darwin 进化，但缺少用户反馈回流 |
